AutoDoc runs as a GitHub Action. Every time a PR opens, it reads the diff and posts a PR summary, ADR draft, and onboarding update — automatically.

Install free → View on GitHub
.github/workflows/autodoc.yml — add and forget
# Add this file. That's the whole install.
name: AutoDoc

on:
  pull_request:
    types: [opened, synchronize, reopened]

jobs:
  autodoc:
    runs-on: ubuntu-latest
    permissions:
      pull-requests: write
      contents: write
    steps:
      - uses: paprika-org/autodoc@v0.1
        with:
          openai_api_key: ${{ secrets.OPENAI_API_KEY }}
          github_token: ${{ secrets.GITHUB_TOKEN }}
How it works

One workflow file. Docs on every PR.

AutoDoc hooks into your existing CI. Nothing to host, nothing to maintain.

01

PR opened

A developer opens or updates a pull request. AutoDoc triggers automatically via GitHub Actions.

02

Diff analysed

AutoDoc reads the full diff, the PR title, and recent commit messages to understand what changed and why.

03

Docs posted

A structured comment appears on the PR: summary, architecture note, and onboarding impact. Already there when reviewers arrive.

04

Saved to repo

ADR drafts and onboarding updates are committed to docs/ so they accumulate over time.

What it actually generates

This was generated from a real merged PR in iFurySt/open-codex-computer-use. AutoDoc ran on the diff with no human input.

PR #19 — fix(cli): accept numeric element_index arguments
AutoDoc comment on iFurySt/open-codex-computer-use

📄 AutoDoc — PR Summary

This PR fixes a bug where CLI arguments for element_index were incorrectly handled when passed as numbers, causing failures in tool commands. A normalisation step now accepts both string and numeric values. Both Linux and Windows versions updated; new test cases added for optionalElementIndex.

Architecture Note

Minor change — no ADR needed. Input type coercion handled at the CLI argument boundary, not inside tool logic.

Onboarding Impact

New contributors: element_index accepts integers or strings. Normalisation happens before any tool call — no need to handle both types downstream.
